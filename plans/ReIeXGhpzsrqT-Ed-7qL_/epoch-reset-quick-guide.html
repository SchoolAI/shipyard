🔄 Epoch Reset - Quick Reference Guide

⚠️ Before You Start

Resetting epochs will clear all data from affected peers. Coordinate with your team before triggering a reset.

When to Use

  • CRDT data corruption spread across multiple devices
  • Need to force all peers to start with clean state
  • Testing/development - want to clear everyone's data

Quick Start (3 Steps)

Step 1: Set Environment Variable

export MINIMUM_EPOCH=2

Increment from current value (1→2, 2→3, etc.). Never decrease.

Step 2: Restart Servers

pnpm dev:all

Both registry and signaling servers will pick up the new MINIMUM_EPOCH.

Step 3: Notify Users

Tell beta users to reload their browsers. They'll be automatically reset.

What Happens

  1. Server rejects connection - Code 4100 (hub) or error message (P2P)
  2. Client clears IndexedDB - Deletes corrupted plan data
  3. Page reloads - Fresh sync from server (which may be empty)
  4. New data created - Plans now have epoch=2

Nuclear Option (Full Reset)

To clear server-side data too:

# Stop servers
Ctrl+C

# Clear server storage
rm -rf ~/.shipyard/plans/

# Restart with bumped epoch
export MINIMUM_EPOCH=2
pnpm dev:all

This gives everyone a completely clean slate.

Verification

Check Server Logs

# Registry server should show:
{"msg":"Rejecting connection: epoch too old","planEpoch":1,"minimumEpoch":2}

# Signaling server should show:
[ValidateEpoch] Rejecting client for plan xxx: epoch 1 < minimum 2

Check Browser

  1. Open DevTools → Console
  2. Look for: [Shipyard] Plan "xxx" rejected: epoch too old
  3. Check Application → IndexedDB → plan databases should be cleared

✅ Success Criteria

  • All peers with epoch=1 are rejected and reload
  • New plans created have epoch=2
  • No corrupted data propagates via hub or P2P
  • System is operational with clean state

Troubleshooting

Infinite Reload Loop

Cause: IndexedDB delete failed but page keeps reloading.

Fix: Close all Shipyard tabs. Delete IndexedDB manually via DevTools. Then reload.

Peers Still Have Old Data

Cause: Server MINIMUM_EPOCH wasn't bumped or server didn't restart.

Fix: Verify echo $MINIMUM_EPOCH shows new value. Restart servers.

P2P Still Syncing Corrupted Data

Cause: Signaling server not updated or offline.

Fix: Ensure signaling server is running with same MINIMUM_EPOCH as registry.
