🔍 Epoch-Based P2P Reset - Review Report

Executive Summary

Overall Assessment: CRITICAL ISSUES FOUND

Two critical design flaws undermine the epoch mechanism. The implementation is technically sound, but the architecture doesn't account for P2P propagation and CRDT mutability.

🚨 Critical Issues (Design Flaws)

Issue 1: WebRTC P2P Does NOT Propagate Epoch Validation

CRITICAL Design gap

Problem: The epoch field syncs via CRDT, but WebRTC peers never validate it. The server only checks epoch on WebSocket connections. Two browsers can sync corrupted data directly via P2P, bypassing the server entirely.

Impact: The entire reset mechanism can be bypassed. A peer with epoch=1 (corrupted) can:

  1. Connect to another peer via WebRTC (signaling doesn't validate epochs)
  2. Sync its corrupted Y.Doc state
  3. The receiving peer now has corrupted data despite MINIMUM_EPOCH=2

Evidence:

// apps/web/src/hooks/useMultiProviderSync.ts:478-549
rtc = getOrCreateWebrtcProvider(roomName, ydoc, signalingServer);
// No epoch validation - just creates provider and starts syncing

Fix Options:

  • Option A: Add browser-side epoch validation after WebRTC sync
  • Option B: Disable WebRTC when hub connection shows epoch mismatch
  • Option C: Make signaling server validate epochs (requires server control)

Issue 2: Epoch Can Be Overwritten Via CRDT Sync

CRITICAL packages/schema/src/yjs-helpers.ts:534

Problem: Epoch is stored in Y.Map as a regular field. It syncs like any other CRDT data using last-writer-wins semantics. A stale peer can overwrite the epoch value during sync.

Impact: A peer with epoch=1 that joins late could write its epoch value to the Y.Doc, overwriting epoch=5 from other peers. This defeats the versioning mechanism.

Evidence:

// yjs-helpers.ts:534
map.set('epoch', init.epoch ?? DEFAULT_EPOCH);
// Just a regular Y.Map field - no special handling

Current behavior:

  1. Server creates plan with epoch=2
  2. Offline peer with old code (epoch=1) reconnects
  3. CRDT sync runs - peer's epoch=1 could win depending on timestamps
  4. All peers now have epoch=1 despite MINIMUM_EPOCH=2

Fix Options:

  • Option A: Epoch should be server-only metadata (not in Y.Doc)
  • Option B: Use Y.Map transactions with conflict resolution (always take max)
  • Option C: Validate epoch BEFORE applying any CRDT updates

⚠️ High Severity Issues

Issue 3: IndexedDB Delete Failure → Infinite Reload Loop

HIGH apps/web/src/utils/epochReset.ts:10-36

Problem: If deletePlanDatabase() fails or times out, it returns false. But handleEpochRejection() ignores this and calls window.location.reload() anyway. Browser reloads, y-indexeddb loads same corrupted data, epoch rejection triggers again.

async function deletePlanDatabase(planId: string): Promise<boolean> {
  const timeout = setTimeout(() => {
    resolve(false);  // Failure returns false, doesn't throw
  }, 5000);
}

export async function handleEpochRejection(planId: string): Promise<void> {
  await deletePlanDatabase(planId);  // Result ignored!
  window.location.reload();  // Reloads even if delete failed
}

Fix: Add sessionStorage flag to track reset attempts. If delete fails twice, show error UI instead of looping.

Issue 4: onblocked Event Not Handled - User Stuck

HIGH apps/web/src/utils/epochReset.ts:28-30

Problem: When other tabs have IndexedDB open, onblocked fires but handler just waits. If connections don't close within 5 seconds, timeout hits and Issue 3 occurs.

Impact: Users with multiple Shipyard tabs get stuck in reload loop.

Fix:

  • Use BroadcastChannel to tell other tabs to close their connections
  • Show UI: "Please close other Shipyard tabs to complete reset"
  • Increase timeout or provide manual retry button

Issue 5: WebSocket Close Handler Race Condition

HIGH apps/web/src/hooks/useMultiProviderSync.ts:428-443

Problem: Close handler is attached in wsStatusListener callback. If the server closes with 4100 before the callback fires, the rejection is missed.

Impact: Epoch rejection could be silently ignored, leaving corrupted data.

Fix: Attach handler immediately after creating WebsocketProvider, with null check.

const wsInternal = ws as WebsocketProvider & WebsocketProviderInternal;

  wsSocket.addEventListener(...);

Standards violation: Project forbids eslint-disable comments.

Fix: Add runtime check before accessing .ws property.

Issue 7: MINIMUM_EPOCH Can Decrease Between Restarts

MEDIUM apps/server/src/config/env/registry.ts

Problem: No validation prevents MINIMUM_EPOCH from decreasing. If admin accidentally sets it lower, previously-reset clients become "valid" again.

Fix: Store previous MINIMUM_EPOCH persistently, validate new value is >= old value.

Issue 8: Missing Epoch in Plan Index

MEDIUM apps/server/src/tools/create-task.ts:259-268

Problem: Plan index doesn't include epoch, so you can't filter stale plans without loading full Y.Docs.

Fix: Add epoch to PlanIndexEntry schema.

Issue 9: No Telemetry for Epoch Rejections

MEDIUM apps/server/src/registry-server.ts:292-304

Problem: Only logger.warn when rejecting. No metrics to track how many rejections occur.

Fix: Add structured telemetry for operations monitoring.

Issue 10: window.location.reload() Can Fail

MEDIUM apps/web/src/utils/epochReset.ts:36

Problem: Reload can be blocked by service workers, beforeunload handlers, or browser extensions.

Impact: Client left with deleted IndexedDB but corrupted in-memory Y.Doc.

Fix: Force navigation or destroy Y.Doc as fallback.

🔧 Recommended Fixes

Priority 1: Address Critical Design Flaws

Two approaches to fix Issues 1 & 2:

Approach A: Server-Authoritative Epoch (Complex)

  • Move epoch out of Y.Doc (server-only metadata)
  • Send epoch in WebSocket handshake, validate before sync
  • Add epoch to WebRTC awareness state, validate on peer connection
  • Prevent epoch from being written to CRDT at all

Approach B: Disable WebRTC During Reset (Simple)

  • When hub detects epoch mismatch, send "disable P2P" signal
  • Client disconnects WebRTC, only syncs via hub
  • After reset completes, re-enable WebRTC
  • Simpler but requires hub coordination

Approach C: Hub-Only Mode for Beta (Simplest)

  • Since this is beta cleanup only, just disable WebRTC globally
  • Set enableWebRTC: false in config
  • All sync goes through hub - epoch validation always happens
  • Re-enable WebRTC after beta phase

Priority 2: Fix High Severity UX Issues

  • Issue 3: Add sessionStorage flag to prevent reload loop
  • Issue 4: Show UI for blocked IndexedDB, use BroadcastChannel
  • Issue 5: Attach close handler immediately, not in callback

Priority 3: Standards Compliance

  • Issue 6: Remove eslint-disable, add runtime check or update biome.json globally
  • Clean up "what" comments to explain "why"
  • Add exhaustive default to message type switch

🎯 Recommendation

For beta cleanup: Use Approach C (disable WebRTC temporarily)

Rationale:

  • Simplest fix - one-line config change
  • Eliminates both critical issues immediately
  • Acceptable for beta (small user base, all local testing)
  • Can re-enable WebRTC after data is clean

For production: Implement Approach A (server-authoritative epoch)

Rationale:

  • Proper solution for distributed system
  • Maintains P2P benefits (offline editing, etc.)
  • Epoch becomes a control plane concern, not data plane
