How It Works Now

Epoch validation happens at BOTH network entry points, preventing corrupted data from spreading via any path:

┌─────────────────────────────────────────────────────────────┐ │ EPOCH VALIDATION POINTS │ ├─────────────────────────────────────────────────────────────┤ │ │ │ Browser │ │ │ │ │ ├──WebSocket──► Registry Server ──► VALIDATES EPOCH │ │ │ (code 4100 if invalid) │ │ │ │ │ └──WebRTC──────► Signaling Server ──► VALIDATES EPOCH │ │ (error message if invalid) │ │ │ │ Corrupted peers (epoch=1) blocked at BOTH entry points │ │ when MINIMUM_EPOCH >= 2 │ │ │ └─────────────────────────────────────────────────────────────┘

How to Trigger a Reset

Step 1: Set MINIMUM_EPOCH

# In terminal (or .env file):
export MINIMUM_EPOCH=2

Step 2: Restart Servers

pnpm dev:all
# Both registry AND signaling servers pick up new MINIMUM_EPOCH

Step 3: What Happens to Each Peer

Connection Type Validation Point Client Behavior
Hub Sync Registry server (WebSocket) Connection closed with code 4100 → clear IndexedDB → reload
P2P Sync Signaling server (WebRTC handshake) Error message received → clear IndexedDB → reload

Step 4: Server-Side Cleanup (Optional)

# For complete reset, clear server storage:
rm -rf ~/.shipyard/plans/
# Then restart: pnpm dev:all

Critical Design Decisions

Why Two Validation Points?

Peers can sync via TWO paths:

  • Path 1: WebSocket to registry server (hub)
  • Path 2: WebRTC P2P via signaling server

Without signaling server validation, corrupted peers could bypass hub checks and spread data via direct P2P connections.

Why Epoch in CRDT Can Still Work

Original concern: Epoch stored in Y.Doc means stale peers could overwrite it.

Mitigation: Validation happens BEFORE peers join the network. If a peer with epoch=1 tries to connect when MINIMUM_EPOCH=2, they're rejected at the signaling server and never get to sync their stale epoch value.

The key is gatekeeper validation - corrupted data never enters the network.

Fallback Behaviors

Scenario Behavior
IndexedDB delete blocked (other tabs open) Timeout after 10s, force navigation to root
Delete fails twice Throw error, stop reload loop
window.location.reload() fails Force navigate to root URL
WebSocket closes before handler attached Handler attached on status event (catches early close)
Signaling server offline WebRTC validation skipped, hub validation still works

Known Limitations

1. Offline Peers Lose Data

If a peer is offline when MINIMUM_EPOCH is bumped, their local data becomes invalid. When they reconnect, they'll be forced to clear and sync fresh.

Impact: Offline work is lost during epoch transitions.

Acceptable for beta: Small user base, infrequent resets.

2. Requires Coordination

MINIMUM_EPOCH must be set consistently on both registry and signaling servers. Mismatched values could allow partial bypass.

Mitigation: Both servers read from same environment variable.

3. No Partial Reset

Epoch is global - can't reset just one plan. Bumping MINIMUM_EPOCH invalidates ALL plans with lower epochs.

Acceptable for beta: Small dataset, can recreate plans easily.

Future Improvements (Post-Beta)

Per-Plan Epochs

Instead of global MINIMUM_EPOCH, validate epoch on a per-plan basis. This allows surgical resets without affecting all plans.

Epoch History Tracking

Log epoch bumps with timestamps and reasons for audit trail.

Graceful Offline Handling

Queue offline changes and prompt user to merge/discard when epoch has changed.

Schema Migrations

Separate schemaVersion from epoch:

  • schemaVersion: Data structure changed → migrate
  • epoch: Data corrupted → discard
