How It Works

Plan-Index Metadata Structure

{
  id: 'plan-index',      // Constant
  createdAt: number,     // Timestamp when created
  updatedAt: number,     // Last update time
  epoch: number          // Version (defaults to 1)
}

Epoch Validation Flow

Browser connects to plan-index
          ↓
Registry server loads plan-index Y.Doc
          ↓
Check if metadata exists
  ├─ NO → Initialize with current MINIMUM_EPOCH
  └─ YES → Validate epoch >= MINIMUM_EPOCH
          ↓
  epoch < MINIMUM_EPOCH?
          ↓
      YES → Close WebSocket (code 4100)
          ↓
   Browser detects rejection
          ↓
   Clear plan-index IndexedDB
          ↓
      Reload page
          ↓
   Sync fresh (empty) plan-index

What Gets Cleared When Plan-Index is Rejected

When plan-index epoch < MINIMUM_EPOCH, the browser clears the entire plan-index IndexedDB database, which includes:

Data Type Impact
PlanIndexEntry objects Sidebar becomes empty (all plan entries cleared)
InputRequests array All pending/answered requests cleared (fixes invalid timeouts)
ViewedBy maps Read/unread status reset (all plans show as unread)
Event-level viewedBy Per-event read status cleared
Navigation state Cross-device routing cleared

Individual plan data is NOT affected - only the index is cleared. Plans still exist in their own Y.Docs.

What This Solves

Original Problem: Invalid Timeouts

InputRequests with old timestamps persisted in plan-index because plan-index had no epoch validation. When MINIMUM_EPOCH was bumped, individual plans were rejected but plan-index continued syncing old data.

Solution

Plan-index now has metadata with epoch field. When MINIMUM_EPOCH is bumped:

  • Individual plans with old epoch → rejected ✅
  • Plan-index with old epoch → rejected ✅ (NEW)
  • Old InputRequests cleared ✅
  • Old PlanIndexEntry objects cleared ✅
  • Complete clean slate across all data ✅
